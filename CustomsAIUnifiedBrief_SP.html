Chapter 01

Product definition
for Clear AI

Scope, user needs, feature direction, success measures, and commercial rollout sequencing for the product we want to build.

Problem Statement

Why this
needs to exist

Saudi customs clearance is largely manual, error-prone, and locked inside carrier-specific tools. Agents and carriers spend significant time on classification tasks that AI can handle — and when they get it wrong, shipments are delayed.

Pain 01
Incomplete product data stalls classification

Importers rarely provide complete product descriptions. Agents must manually research HS codes for vague inputs — a slow, error-prone process that directly delays clearance at Saudi borders.

Pain 02
Manual classification is a bottleneck at scale

Carriers processing high shipment volumes have no scalable way to classify goods automatically. Every item requires human effort, creating a linear relationship between volume and cost.

Pain 03
Existing tools are siloed to single carriers

Automation tools that do exist are internal to specific logistics providers. Independent clearance agents — who serve multiple carriers and importers — have no access to intelligent tooling.

Pain 04
ZATCA compliance adds layers of complexity

Saudi-specific requirements — Arabic descriptions, ZATCA XML format, and additional regulatory certificates — are not handled by generic global classification tools, forcing manual compliance work on every shipment.

Goals

What success
looks like

Goals are split between what users gain and what the product needs to prove at MVP. Each goal is measurable — not a direction, but an outcome.

User Goals
  • Clearance agents can classify a product and generate a ZATCA-ready draft in under 30 seconds
  • Carriers can submit bulk manifests and receive classified outputs without manual intervention per item
  • Low-confidence results are surfaced clearly so human review effort goes only where it's needed
  • Any user — carrier or independent agent — can access the same capability without carrier lock-in
Product Goals
  • Demonstrate that AI classification meets or exceeds manual accuracy on a real Saudi shipment dataset
  • Ship a working API that a carrier integration can go live on within the MVP window
  • Cover all P0 features — including bulk processing and declaration generation — in V1
  • Establish ZATCA direct submission as a validated path for V2, not an assumption

📌
Non-Goals (V1): Clear AI will not handle duty payment, broker licensing, physical customs inspection coordination, or outbound export declarations in this release.
✕

Duty payment processing

Clear AI provides duty rate references, not payment facilitation. Payment flows through existing banking or carrier systems.

✕

Broker licensing or agent credentialing

The platform does not verify or manage the legal status of clearance agents. That is an identity and compliance concern outside MVP scope.

✕

Export declarations

V1 covers import clearance only. Outbound export flows are architecturally supported but not in scope.

✕

GCC cross-border expansion

Saudi customs rules only. Each GCC country has meaningfully different requirements — expansion is a later initiative.

Level 3 — HS Code Resolution (High Value only)

How Naqel resolves the HS codes

Each item needs a valid 12-digit ZATCA tariff code. Clients submit codes of varying quality — findings below are based on analysis of 353,623 line items across 31,017 waybills [date range not recorded — confirm with Naqel team].

Clean the code
Remove dots, spaces, and any non-numeric characters. Some merchants send codes like 610.821.00 — these become 61082100.
If shorter than 12 digits, build search keys
Strip one digit from the right, repeatedly, down to 4 digits. Each shorter version becomes a candidate key to search against the master list.
61082100 → 6108210 610821 61082 6108
Look up each candidate in the ZATCA master list
The system searches the HSCodeMaster table for each candidate key. It returns all 12-digit ZATCA codes that match, along with their Arabic description, duty rate, and a flag for how unit cost should be calculated.
Pick the best match
From all results, prefer the one that matched the longest candidate key — meaning the client's original code was the most specific. If two results match equally, prefer the shorter ZATCA code (more general category).
Use the top result
The winning 12-digit code becomes the tariff code sent to ZATCA. Its Arabic name becomes the goods description. Non-Arabic characters are stripped from the description before submission.
⚠️
Arabic description rule: ZATCA rejects descriptions that word-for-word match the official tariff text. Workaround: extra words are appended manually before submission.

⚠ When multiple candidates exist
If the client had sent 62046200, the lookup returns 5 valid ZATCA codes — casual trousers, trousers with vest, overalls, leggings, short pants — all under the same prefix, all 5% duty. The system picks the first one in table order. It has no way to know which is correct without reading the item description.
✓ Complete code — no algorithm needed
If the client sends a full 12-digit code like 851713000000 (smartphones, exempted), the system uses it directly. Step 1 only runs to strip any dots — no search needed.
Current system
① Complete code (83.9%) — normalised, passed through as-is. No validation against item description.
② Short/incomplete code (13.6%) — prefix algorithm finds candidates in HSCodeMaster, picks the first match by table order. Item description is ignored. If 5 valid leaf codes share the same prefix, the algorithm always picks the first — regardless of what the item actually is.
③ Missing code (~1.8% HV) — gateway team manually derives a code and submits via ZATCA web portal. Corrected code is never written back upstream.
In all three cases the system has no way to verify whether the resolved code actually matches what the item description says.
Where AI fits
①
Disambiguation — client sends 62046200, system finds 5 leaf codes (trousers, shorts, overalls, leggings…). AI reads the item description and selects the correct one instead of defaulting to table order.
②
Classification from scratch — no HS code provided at all. AI classifies the item using its description and category, eliminating the manual gateway portal workaround entirely.
③
Pre-submission validation — client provides a complete, valid-format code. AI cross-checks it against the item description and flags mismatches before ZATCA submission — catching the silent errors the current system cannot detect.
Layers ② and ③ are entirely unaddressed by the current rule-based system. Layer ① is partially addressed but description-blind.

Known Issues & Edge Cases

Where the process
can break down

Six confirmed issues, sourced from the March 8 and March 30 meeting notes and sample analysis. The most operationally significant is the missing-HS workaround — which creates a data split between the system and what's actually in ZATCA.

🔴 Critical — Data Gap
Missing HS Code — ZATCA Portal Bypass
Estimated rate: ~1–2% of HV shipments
A shipment arrives with no HS code. The gateway team manually derives a code from the goods description and submits it directly through the ZATCA web portal — bypassing the standard Infotrack system flow entirely.
Actual handling (from 30 Mar meeting): The "corrected" HS code lives in ZATCA/Bayan only — it is not written back upstream into Infotrack or the data pipeline. This means your training data will have a systematic gap: the cases that needed correction are exactly the hardest cases, and they're missing from the upstream record.
🟣 Blind Spot
Wrong HS Code (Valid Format)
Estimated rate: unknown — not measured today
Client submits a 12-digit code that is syntactically correct but categorises the wrong product (e.g. phone code submitted for clothing).
Current handling: Not detected. The system accepts any valid-format code. No cross-validation against item description exists. This is the highest-value AI automation opportunity — and the hardest to baseline without labelled ground truth.
🟠 Medium Risk
Description vs. HS Code Mismatch
Frequency: visible in sample data
The resolved Arabic description doesn't match what the client actually shipped (e.g. "Dresses" mapped to Trousers). Also seen in sample data: many descriptions are generic like "غيرها" (other/misc).
Current handling: ZATCA does not auto-reject. Customs staff may manually flag — higher risk on HV shipments. No automated description validation exists today.
🟠 Medium Risk
ZATCA Exact-Match Description Rejection
Trigger: when description = HS tariff text verbatim
ZATCA actively rejects item descriptions that are an exact string match to the HS code's official tariff text.
Current workaround: Extra words are manually appended to the Arabic description before submission. This is a convention, not a system rule — it can silently break if whoever maintains the process changes.
🟢 Auto-Handled
Incomplete HS Code (<6 digits)
Estimated rate: ~5–10% of HV shipments
Client submits a code shorter than 6 digits (or with formatting like dots). Common with certain Chinese merchant categories.
Current handling: Automatically resolved via the 5-step prefix-traversal algorithm + HSCodeMaster lookup. Chinese merchants sending full 12-digit codes is common — so the true problem volume may be concentrated in a few merchant segments.
🟢 Low Risk Today
Hardcoded ZIP & PO Box Placeholders
Affects: all submissions
All declarations use placeholder values: zipCode=1111, poBox=11. These are not real addresses.
Current handling: ZATCA does not currently validate these fields strictly. Low risk today — but a latent fragility. If ZATCA tightens validation, all submissions fail until real values are implemented system-wide.
